Langage de structure

HTML

HTML

HyperText Markup Language · Structure des pages web

3
Haute
1.1

🖱️ Clickjacking

Explication

Le Clickjacking superpose une page malveillante sur une page légitime via une <iframe> rendue quasi-invisible. L'utilisateur croit interagir avec le site de l'attaquant mais déclenche des actions sur le site légitime — virement bancaire, suppression de compte, validation de commande.

Glissez le curseur pour révéler l'iframe cachée. Observez comment la page malveillante (visible) recouvre la banque (en dessous).

🏦 MaBanque.fr
Confirmer le virement de 5 000 €
🎉 Félicitations !
Vous avez gagné un prix exclusif
88%
En attente d'un clic…
Parade
  • En-tête HTTP X-Frame-Options: DENY sur le serveur
  • CSP : Content-Security-Policy: frame-ancestors 'none'
  • Framebuster JavaScript (vérification window.top)
PHP — en-têtes
header('X-Frame-Options: DENY');
header("Content-Security-Policy: frame-ancestors 'none'");
Moyenne
1.2

📝 Formulaires non sécurisés

Explication

Les formulaires en HTTP transmettent les données en clair sur le réseau — interceptables via une attaque Man-in-the-Middle (MITM). L'absence de validation des attributs (type, maxlength, pattern) permet d'envoyer des données inattendues au serveur.

Parade
  • HTTPS obligatoire (certificat TLS)
  • type="password" pour les mots de passe
  • Attributs required, maxlength, pattern
  • Validation côté serveur — la validation HTML est contournable
Haute
1.3

👁️ Données sensibles dans le source

Explication

Le HTML est entièrement lisible via Ctrl+U. Clés API, identifiants, commentaires de debug laissés dans le code sont exploitables directement. Les <input type="hidden"> sont visibles et modifiables depuis la console du navigateur.

Parade
  • Secrets dans des variables d'environnement côté serveur
  • Ne jamais confier la sécurité à un champ caché — toujours vérifier côté serveur
  • Linter de sécurité avant tout déploiement

Critique
2.1

⚡ XSS — Cross-Site Scripting

Explication

Le XSS injecte du JavaScript malveillant dans les pages d'autres visiteurs. Un attaquant peut voler des cookies de session, rediriger vers un site de phishing, ou enregistrer les frappes clavier. La cause : affichage de contenu utilisateur via innerHTML sans filtrage.

🚨 Script détecté ! En production, ce code s'exécuterait chez chaque visiteur du site.
JavaScript — sécurisé
// ❌ DANGEREUX — interprète le HTML :
element.innerHTML = userInput;

// ✅ SÉCURISÉ — traite comme du texte brut :
element.textContent = userInput;

// ✅ Encoder avant affichage :
function escape(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g,  '&quot;');
}
  • Utiliser textContent au lieu de innerHTML
  • CSP header : script-src 'self'
  • Cookies avec flag HttpOnly
Haute
2.2

🗄️ LocalStorage non sécurisé

Explication

Tout code JavaScript de la page peut lire localStorage. En cas de XSS, les tokens, rôles ou données sensibles qui y sont stockés sont immédiatement compromis. La logique de sécurité côté client est contournable depuis la console.

Parade
  • Cookies HttpOnly + Secure + SameSite=Strict
  • Vérifier rôles et droits côté serveur uniquement
  • Ne rien stocker de sensible dans localStorage
Haute
2.3

🔄 CSRF — Cross-Site Request Forgery

Explication

Le CSRF force le navigateur d'un utilisateur connecté à envoyer une requête non voulue vers un site où il est authentifié. Le navigateur joint automatiquement les cookies de session — le serveur pense que la requête est légitime.

Langage côté serveur

PHP

PHP

Hypertext Preprocessor · Langage serveur dynamique

3
Critique
3.1

📂 Inclusion de fichiers (LFI / RFI)

Explication

LFI (Local File Inclusion) : lecture de fichiers locaux arbitraires via l'URL. RFI (Remote File Inclusion) : inclusion d'un fichier distant contrôlé par l'attaquant menant à une exécution de code à distance (RCE). Cause : paramètre passé directement à include().

🎮Démo — Simulateur LFILive

Modifiez le paramètre ?page= et observez ce que le serveur «inclurait». Essayez les payloads pré-définis.

site.fr/index.php?page=
Résultat d'inclusion…
Parade
  • Liste blanche des fichiers autorisés
  • Désactiver allow_url_include dans php.ini
  • Jamais de paramètre utilisateur dans include()
PHP — liste blanche
$pages_ok = ['accueil', 'contact', 'about'];
$page    = $_GET['page'] ?? 'accueil';
if (in_array($page, $pages_ok)) {
    include('pages/' . $page . '.php');
} else {
    http_response_code(404);
}
Critique
3.2

📤 Upload de fichiers malveillants

Explication

Sans vérification stricte, un attaquant peut uploader un fichier PHP déguisé en image. Placé dans un dossier web accessible, il devient un webshell permettant l'exécution de n'importe quelle commande sur le serveur.

Parade
  • Vérifier le type MIME réel avec finfo_file() (pas le Content-Type)
  • Renommer les fichiers uploadés (nom aléatoire)
  • Stocker les uploads hors de la racine web
  • Extension bloquée : jamais .php, .phtml, .phar…
Moyenne
3.3

⚠️ Exposition des erreurs PHP

Explication

Des messages d'erreur détaillés en production révèlent : chemins internes, noms de tables SQL, version PHP, structure du code. Ces informations facilitent considérablement la cartographie du système par un attaquant.

Parade
  • display_errors = Off en production
  • log_errors = On vers un fichier sécurisé inaccessible depuis le web
  • Pages d'erreur personnalisées sans détails techniques

4.2

🔑 Mots de passe en clair & hashage faible

Explication

Stocker les mots de passe en clair ou via MD5/SHA1 est critique. Ces algorithmes sont cassés pour les mots de passe — des rainbow tables permettent de retrouver le clair en quelques secondes sur des sites comme crackstation.net.

4.3

🔓 Excès de privilèges MySQL

Explication

Si l'application utilise le compte root, une SQLi réussie peut supprimer toutes les tables, lire des fichiers système (LOAD_FILE) ou créer des backdoors MySQL avec GRANT ALL.

  • Compte dédié : SELECT, INSERT, UPDATE uniquement
  • Fermer le port 3306 via pare-feu
  • Supprimer les comptes MySQL inutilisés

Synthèse · TP Cybersécurité

Ne jamais faire
confiance au client.

Valider systématiquement côté serveur, quelle que soit la validation côté client. Chaque technologie présente ses propres vulnérabilités — toutes partagent la même origine : un manque de filtrage ou de cloisonnement.

La défense efficace repose sur une approche en couches (défense en profondeur). Une seule faille non corrigée peut compromettre un système entier.
