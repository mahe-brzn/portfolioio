Langage de structure

HTML

HTML

HyperText Markup Language · Structure des pages web

Haute
1.1

🖱️ Clickjacking

Explication
Le Clickjacking superpose une page malveillante sur une page légitime via une <iframe> transparente. L'utilisateur clique sur le site malveillant mais déclenche en réalité des actions sur le site légitime (virement, suppression de compte…).

Parade
  • En-tête HTTP X-Frame-Options: DENY
  • CSP : frame-ancestors 'none'
  • Vérification JavaScript (framebuster)
Moyenne
1.2

📝 Formulaires non sécurisés

Explication
Formulaires en HTTP (non HTTPS) transmettent les données en clair — interceptables via attaque MITM. L'absence de validation des attributs (type, maxlength) permet d'envoyer des données inattendues.
Parade
  • HTTPS (certificat TLS obligatoire)
  • type="password" pour les mots de passe
  • Attributs required, maxlength, pattern
  • Validation côté serveur (la validation HTML est contournable)
Haute
1.3

👁️ Données sensibles dans le code source

Explication
Le HTML est entièrement lisible via Ctrl+U. Clés API, identifiants, commentaires de debug laissés dans le code sont directement exploitables. Les <input type="hidden"> sont modifiables depuis la console.
Parade
  • Secrets en variables d'environnement côté serveur
  • Ne jamais confier la sécurité à un champ caché
  • Linter de sécurité avant déploiement

Langage côté client

JS

JavaScript

Langage de script exécuté dans le navigateur

Critique
2.1

⚡ XSS — Cross-Site Scripting

Explication
Le XSS injecte du JavaScript malveillant dans les pages d'autres utilisateurs. Un attaquant peut voler des cookies de session, rediriger, ou enregistrer les frappes clavier. Cause : affichage de contenu utilisateur sans filtrage.

JavaScript — sécurisé
// ❌ DANGEREUX — interprète le HTML :
element.innerHTML = userInput;

// ✅ SÉCURISÉ — traite comme du texte brut :
element.textContent = userInput;

// ✅ Encoder avant affichage :
function escape(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
  • Utiliser textContent au lieu de innerHTML
  • CSP header : script-src 'self'
  • Cookies avec flag HttpOnly
Haute
2.2

🗄️ LocalStorage non sécurisé

Explication
Le code JS est lisible via F12. Les données dans localStorage sont accessibles par tout JS de la page — en cas de XSS, elles sont immédiatement compromises. Ne jamais y stocker tokens, rôles ou données sensibles.

  • Cookies HttpOnly + Secure + SameSite=Strict
  • Vérifier rôles/droits côté serveur uniquement
  • Ne rien stocker de sensible en localStorage
Haute
2.3

🔄 CSRF — Cross-Site Request Forgery

Explication
Le CSRF force le navigateur d'un utilisateur connecté à effectuer des requêtes non voulues. L'attaquant envoie un lien piégé — le navigateur de la victime envoie automatiquement ses cookies de session.

Langage côté serveur

PHP

PHP

Hypertext Preprocessor · Langage serveur dynamique

Critique
3.1

📂 Inclusion de fichiers (LFI / RFI)

Explication
LFI : lecture de fichiers locaux arbitraires via l'URL. RFI : inclusion d'un fichier distant contrôlé par l'attaquant → exécution de code (RCE). Cause : paramètre passé directement à include().
🎮Démo — Simulateur LFILive

Modifiez le paramètre ?page= et voyez ce que le serveur «inclurait».

site.fr/index.php?page=
Résultat d'inclusion…
Parade
  • Liste blanche des fichiers autorisés
  • Désactiver allow_url_include dans php.ini
  • Jamais de paramètre utilisateur dans include()
Critique
3.2

📤 Upload de fichiers malveillants

Explication
Sans vérification, un attaquant uploade un fichier PHP déguisé en image. Placé dans un dossier web accessible, il devient un webshell permettant l'exécution de commandes sur le serveur.
Parade
  • Vérifier MIME réel avec finfo_file()
  • Renommer les fichiers uploadés
  • Stocker hors de la racine web
Moyenne
3.3

⚠️ Exposition des erreurs PHP

Explication
Des messages d'erreur détaillés en production révèlent : chemins internes, noms de tables, version PHP. Ces informations facilitent la cartographie du système.
Parade
  • display_errors = Off en production
  • log_errors = On vers fichier sécurisé
  • Pages d'erreur sans détails techniques

4.2

🔑 Mots de passe en clair & hashage faible

Explication
Stocker les mots de passe en clair ou avec MD5/SHA1 est critique. MD5 et SHA1 sont cassés — des rainbow tables permettent de retrouver le mot de passe en quelques secondes.

Synthèse · TP Cybersécurité

Ne jamais faire
confiance au client.

Valider systématiquement côté serveur, quelle que soit la validation côté client. Chaque technologie présente ses propres vulnérabilités mais elles partagent la même origine : un manque de filtrage ou de cloisonnement.

La défense efficace repose sur une approche en couches (défense en profondeur). Une seule faille non corrigée peut compromettre tout un système.
